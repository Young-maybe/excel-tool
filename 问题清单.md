## 问题清单（经验记录）

### 1) PyQt6 报错：ImportError: DLL load failed while importing QtCore
- **现象**：启动时报 `ImportError: DLL load failed while importing QtCore: 找不到指定的程序。`
- **原因**：PyQt6/Qt6 二进制依赖与本机环境/版本组合不匹配（仅加 PATH 不一定能解决）。
- **处理**：将 PyQt6 从 `6.6.0` 强制重装到 `6.7.1` 后恢复正常；并在 `main.py` 中保留异常捕获与日志，避免直接崩溃。

### 2) 多步骤模块的“输入目录/输出目录”不一致导致“未找到文件”
- **现象**：步骤1生成的文件能成功，但步骤2/3提示“未找到对应文件”。
- **原因**：步骤1输出写到程序统一输出目录，但步骤2/3仍从输入目录找上一步产物。
- **处理**：保持“中间强逻辑”不变，只调整 I/O：
  - 需要读取上一步产物的步骤，从 `output_dir` 读取；
  - 模板/原始数据从 `input_dir` 读取；
  - 所有生成文件统一写入 `output_dir`。

### 3) PowerShell 兼容性：命令连接与 heredoc
- **现象**：`cd /d ... && ...`、`python - <<PY` 在 PowerShell 下会报解析错误。
- **处理**：PowerShell 里使用 `;` 连接命令；复杂脚本用 `python -c` 或单独 `.py` 文件执行。

### 4) WinError 32：文件被占用导致复制/写入失败
- **现象**：弹窗提示“处理失败”，日志中出现 `PermissionError: [WinError 32] 另一个程序正在使用此文件，进程无法访问。`
- **原因**：源文件或输出目录同名文件被 Excel/WPS 打开占用；或输入目录存在 `~$` 开头的临时锁文件。
- **处理**：
  - 关闭占用文件的 Excel/WPS 后重试；
  - 自动跳过 `~$` 临时文件；
  - 输出目录若有同名文件则自动改为 `*_处理副本N.xlsx` 避免冲突。

### 5) 大 CSV（例如 40MB）处理很慢
- **现象**：能跑完但耗时很长，尤其是写 Excel 阶段。
- **原因**：
  - `openpyxl` 逐格写入（双层 for 循环）在大数据量下非常慢；
  - `DataFrame.iterrows()` 逐行计算时效也会放大 Python 开销。
- **处理**：
  - 计算部分改为 pandas 向量化（`to_datetime(errors='coerce')` + 条件掩码）；
  - 写 Excel 改用 `xlsxwriter` 批量写入（保留列宽与指定表头黄色背景），通常能显著提速；
  - 读取 CSV 先读表头再用 `usecols` 只读必要列，减少 I/O 与内存压力。

### 6) “可视化百分比进度 + 处理状态提示”如何统一接入（不影响模块逻辑）
- **目标**：所有耗时模块都能显示：
  - 百分比进度条（0~100）
  - 当前阶段状态文本 + 详细日志滚动
  - UI 不冻结（后台线程跑任务）
- **通用做法**：
  - UI 层（`ui/components/folder_task_page.py`、`ui/components/file_task_page.py`）统一嵌入 `AppleProgressPanel`；
  - 点击“开始处理”后，通过 `ui/components/task_runner.py` 把 handler 放进 `QThread` 执行；
  - UI 自动把 `progress_cb(percent, status)` 注入到业务函数（如果业务函数不支持该参数，会自动回退为无进度模式，不影响功能）。
- **业务层改动原则**：
  - 只在关键阶段插入 `if progress_cb: progress_cb(x, "xxx")` 打点；
  - 不修改原有输入输出与核心计算逻辑；
  - 避免在高频循环里每行都上报（会拖慢），建议“每 N 行 / 每块 / 每文件”上报一次。

### 7) 库存报表追加“商品库存导出”sheet：数字变成文本
- **现象**：输出文件中 `商品库存导出` sheet 的部分“数量/金额/库存”等列变成文本，影响排序/求和。
- **原因**：读取 CSV 时 `dtype=str` 后直接写入 openpyxl，Excel 会把整列当文本。
- **处理**：在 `ui/components/stock_tool_legacy_main.py` 追加该 sheet 时做“方向性数值转换”：
  - **编码/条码/SKU/名称等**保持文本
  - **数量/库存/金额/成本/单价等**写为 `int/float` 并设置 number_format（整数 `0` / 小数 `0.00`）

### 8) 飞书连接“看似成功但仍未连接”
- **现象**：点击“连接飞书多维表格”，提示 token 获取成功，但 UI 仍显示未连接、按钮不可用。
- **原因**：requests 分支拿到 token 后没有把 `feishu_connected` 置为 True（仍走旧判定）。
- **处理**：在 `connect_feishu()` 的 requests 分支里，token 成功后立刻：
  - `self.feishu_connected = True`，更新状态标签，并 `return` 结束，避免继续掉到 `lark_oapi` 分支。

### 9) 飞书依赖缺失导致导入失败
- **现象**：报错 `No module named 'lark_oapi'` / `No module named 'requests'` / `No module named 'lark'`。
- **处理**：补齐 `requirements.txt`：`requests`、`lark-oapi`、`lark`，并在 venv 执行 `pip install -r requirements.txt`。

### 10) 文件夹选择弹窗无法做主题适配
- **现象**：Windows 原生“选择文件夹”弹窗无法跟随 QSS（颜色/按钮风格不统一）。
- **原因**：原生对话框由系统绘制，不受 Qt 样式表完整控制。
- **处理**：统一使用 Qt 非原生对话框（`DontUseNativeDialog`），并：
  - 默认打开桌面
  - 左侧快捷栏加入“桌面/文档/下载/用户目录”
  - 相关实现：`ui/components/dialogs.py`

### 11) 使用统计（节省时长）需要可持久化且不影响主流程
- **要点**：
  - 统计失败不能影响任务处理
  - 数据需本地持久化（不随关闭重置）
- **处理**：
  - `services/usage_stats.py` 负责存储与聚合（写入 `resources/usage_stats.json`）
  - `ui/components/usage_stats_page.py` 展示二级页面报表（可返回）
  - 埋点：通用任务页成功回调记录；库存工具三项完成回调记录

### 12) 操作手册写死在代码里，不利于维护
- **现象**：手册内容写死在 `ui/main_window.py` 的常量里，改文案必须改代码。
- **处理**：迁移到 `huancuntxt/*.txt`，运行时通过 `ui/components/manual_text_repo.py` 读取。

### 13) “Undo all” 后找不到入口（不可一键回退）
- **现象**：点了 Undo All 后 `main.py`/UI/模块文件丢失或错乱。
- **原因**：工程不是 Git 仓库时，没有可用的版本回退点。
- **建议**：后续建议把工程初始化为 Git，并提交关键节点，避免同类事故。

### 14) 打包“一键脚本”在 Windows 上闪退/找不到 ps1
- **现象**：双击运行 `.ps1` / `.bat` 后窗口一闪而过，或提示 `-File` 参数对应的 `.ps1` 不存在。
- **原因**：cmd/PowerShell 在某些系统区域/编码下会把**中文脚本名**解析成乱码，导致路径失效。
- **处理**：
  - 提供英文入口：`resources/installer_tools/build_installer.bat` + `build_installer.ps1`
  - 中文入口 `一键生成安装包.bat` 内部转调英文脚本
  - `.bat` 设置 `chcp 65001`，尽量减轻控制台乱码

### 15) PyArmor/加固链路踩坑（与 Python 版本强相关）
- **现象**：
  - `python -m pyarmor` 报 `No module named pyarmor.__main__`
  - PyArmor 9.x 没有 `pack` 子命令
  - `pyarmor-7 pack` 报 `Python 3.11+ is not supported now`
- **原因**：
  - PyArmor 9.x CLI 变化较大，且 legacy 的 `pack` 需要 `pyarmor-7`
  - legacy pack 目前只支持到 Python 3.10
- **处理**：
  - `requirements-build.txt` 固定 PyArmor 版本，并使用 `pyarmor-7.exe` 执行 pack
  - 脚本检测当前 Python 版本：
    - 若可用 `py -0p` 检测到 Python 3.10，则自动创建 `.venv_build310` 作为“打包专用环境”来启用加固
    - 否则自动关闭加固，回退到纯 PyInstaller 打包（保证先能出安装包）

### 16) PyInstaller 在 PowerShell 下参数丢失（scriptname required）
- **现象**：`pyinstaller: error: the following arguments are required: scriptname`
- **原因**：PowerShell 内置自动变量 `$Args` 会吞掉/覆盖脚本里自定义的 `$Args`，导致参数传递异常。
- **处理**：
  - 避免使用 `$Args` 变量名，统一改为 `$PyInstallerArgs`
  - 直接调用 venv 下的 `pyinstaller.exe`（而不是 `python -m PyInstaller`）保证参数一致

### 17) Inno Setup 编译 installer.iss 连续报错
- **现象**：
  - `AppId` 大括号/常量解析错误
  - `ChineseSimplified.isl` 缺失导致编译中断
- **原因**：
  - Inno Setup 的 `{...}` 是常量语法，`AppId` 应使用 GUID 且需正确转义大括号
  - 部分 Inno Setup 安装不包含 `Languages\\ChineseSimplified.isl`
- **处理**：
  - `installer.iss` 改用 GUID 形式的 `AppId`
  - 语言包固定使用 `compiler:Default.isl`，确保“必定能编译出安装包”

### 18) 打包产物/路径乱码（Excel澶勭悊…）
- **现象**：`dist/`、`build/`、zip 文件名出现乱码，影响后续脚本定位与安装包引用。
- **原因**：Windows 控制台/PowerShell 5.1 对中文输出/文件名在某些情况下会产生 mojibake，且 PyInstaller 默认会用 `--name` 作为目录名。
- **处理**：
  - 构建名统一使用英文：`ExcelTools`
  - 安装包显示名仍保持中文：`Excel处理工具`
  - 输出文件名固定：`Install_ExcelTools.exe`、`Portable_ExcelTools.zip`

### 19) 打包后产生大量中间产物需要清理
- **现象**：`build/`、`dist/`、`*.spec`、打包日志、下载缓存等体积很大，影响工程整洁。
- **处理**：提供清理脚本（保留最终交付物）：
  - `resources/installer_tools/清理打包中间产物.bat`
  - `resources/installer_tools/清理打包中间产物.ps1`
